## 1. Handover from previous chat (starting state)

CASSIA is a multi-user, chat-based accounting assistant built on a hybrid routing core: every question is classified to **SQL** (structured accounting data), **RAG** (IRS Publications + uploaded PDFs), **BOTH**, or **CORE_RECALL** (the user's saved "core" findings, recalled semantically across sessions). Charts render via Plotly. Demo data lives in `accounting.db` (7 tables); per-session uploads attach as a `user_data` namespace; IRS documents live in a Chroma vector store.

**Working method this cycle:** read-only *locator* scripts map the live code → surgical *applier* scripts patch it (backup + idempotent + syntax-checked + auto-restore on failure). Scripts run locally and are verified before anything ships.

Coming into this session the v1 stabilization slate (Section 2) was already applied and pushed (commit `c3dbbbc`), Core had been reset to a clean slate, and the v2 demo fixtures were built. **Open item:** Sim 4 (the quarterly-continuity finale) was failing on forecast / variance queries; root cause not yet diagnosed.

## 4. Patches applied this session (v2)

### P1c — chart numeric detection
`apply_p1c_chart_numeric.py` → `backend/static/index.html` · backup `index.html.pre-p1c-<ts>.bak`
Fixed dates being plotted as the year 2026. Added `_p1IsNumericVal(v)` so a value is treated as numeric only when the whole string parses as a number, replacing loose `parseFloat` checks.

### P1d — chart axis isolation
`apply_p1d_chart_axis_isolation.py` → `backend/static/index.html` · backup `index.html.pre-p1d-<ts>.bak`
Fixed an empty-chart regression where category (vendor) bar charts rendered blank after P1c.
**Root cause:** `renderChart` built its layout with a shallow spread `{...PLOTLY_LAYOUT}`, so every chart shared the same nested `xaxis` object; Plotly mutates layout in place and wrote `xaxis.type='date'`, which then leaked onto the next chart and collapsed text-label charts.
**Fix:** deep-clone the layout (`JSON.parse(JSON.stringify(...))`) and set `xaxis.type` explicitly to `'date'` or `'category'` per chart. Verified across all subsequent date and category charts.

### Router table-awareness
`apply_router_table_awareness.py` → `backend/routers/query_router.py` · backup `query_router.py.pre-routertables-<ts>.bak`
**Root cause:** the router was PDF-aware (it biases data questions toward RAG when a PDF is in the session) but blind to uploaded *data tables*. With a PDF present, "show the forecast from the file I uploaded" routed to RAG → "not found in docs," even though the SQL pipeline already attaches the uploaded table and knows its schema (`_get_user_tables_schema`, `ATTACH … AS user_data`).
**Fix (one file):** added `_get_session_table_names()` (mirrors the existing PDF helper, filters `target='sql'`); taught `route_with_llm` to list those tables and apply explicit precedence — document/policy questions stay RAG, while "show / plot / compare a forecast or figures from an uploaded file" goes SQL — and wired it through `classify_question`. Idempotent, syntax-checked, reversible.
**Confirmed:** the forecast query now routes SQL (`SELECT service_line, q1_2026 FROM user_data.revenue_forecast_2026`) and renders.

### Demo operating notes
- Phrase Core recalls by **subject** ("AR collection risk"), not by meta-recall language ("from my core" / "what I saved"). Trigger words force the Core route but dilute the semantic match and can drop it below the 0.35 relevance threshold.
- Year-to-date book revenue and the Q1 forecast are different scopes — present them as two views, not a single head-to-head number.
- RAG "not found in docs" on out-of-corpus questions is an intentional honesty beat, not a failure.

### Known limitations / post-deploy backlog
- Computed cross-table variance (books ↔ uploaded forecast) is brittle (join-key name mismatch: "Consulting Revenue" vs "Consulting"); demo narrates the side-by-side. Real fix: name normalization or a synthesis route.
- A Core-recall miss falls through to a SQL "no data" stub; it should report "nothing in core matches" instead.
- No blend route to combine recalled saves + live data in one answer (Sim 4 P8 limitation).
- "I have related saves but…" hedge fires at modest relevance — synthesis-prompt tuning.
- Consolidate P1 / P1b / P1c / P1d chart-filter duplication; remove two console-only `[P1 DEBUG]` prints in `chart_builder.py`; add a save-button junk guard.
